# ۱. معرفی کوتاه پروژه

پروژه Castaminofen یک سوپر اپ رسانه‌ای فارسی است که مصرف پادکست، کتاب صوتی و ویدیو را در یک تجربه واحد طراحی کرده است. این پروژه هم برای وب (`apps/web`) و هم برای موبایل (`apps/mobile`) پیاده‌سازی شده و از معماری monorepo با `pnpm` و `turbo` استفاده می‌کند.

هدف پروژه فراهم کردن تجربه mobile-first با RTL و dark mode پیش‌فرض، ناوبری ساده، پخش رسانه و پنل سازنده اولیه است. همچنین هدف کلی این است که مصرف محتوا و تولید محتوا در یک بستر هم‌خوان و فارسی‌محور عرضه شود.

وضعیت فعلی پروژه: UI پایه و معماری کلی موجود است، صفحات اصلی وب و موبایل ساخته شده‌اند، اما برخی قاب‌های UX هنوز در حالت placeholder یا stub باقی مانده‌اند. بخش پرداخت/پریمیوم، دانلود آفلاین موبایل و برخی جریان‌های سازنده به صورت ناقص یا نیمه‌کاره دیده می‌شوند.

---

# ۲. خلاصه وضعیت رابط کاربری

- کیفیت کلی UI: خوب و قابل قبول برای یک MVP. رنگ‌بندی برند مشخص است و RTL به صورت بنیادی رعایت شده.
- میزان پیشرفت: بخش وب به صورت تقریباً کامل‌تری پیاده شده است. موبایل دارای UI اصلی است اما چند صفحه واضحاً placeholder دارد.
- نقاط قوت:
  - پالت رنگی بنفش/سبز و تم تاریک هماهنگ با docs/UI-UX.
  - طراحی RTL در وب و موبایل.
  - استفاده از فونت Vazirmatn و رعایت تایپوگرافی فارسی.
  - معماری monorepo و shared package برای توکن‌ها و نوع‌ها.
  - وجود state management مشترک با Zustand.
- نقاط ضعف:
  - سیستم طراحی به صورت کامل یکپارچه نشده و web/mobile هنوز کاملاً همگرا نیستند.
  - placeholderهای مشخص در `Downloads` موبایل، `Tools` موبایل و بخشی از تجربه ویدیو.
  - برخی بخش‌های سازنده و پرداخت هنوز حالت stub دارند.
  - نبود یک کتابخانه کامپوننت رسمی و کاملاً reusable.

# ۳. معماری Frontend

- Framework: `Next.js 15` برای وب و `React Native` + `Expo` برای موبایل.
- زبان برنامه‌نویسی: `TypeScript` در تمام اپلیکیشن‌ها.
- Build Tool: `pnpm` + `turbo` (monorepo). اسکریپت‌ها در `package.json` ریشه و هر پکیج تعریف شده‌اند.
- Routing: `Next.js App Router` در وب و `Expo Router` در موبایل.
- State Management: `zustand` در وب (`apps/web/src/store/player.ts`) و موبایل (`apps/mobile/store/player.ts`).
- Styling System: وب با CSS variables و `globals.css` استایل‌دهی شده. موبایل از `StyleSheet`/inline styles و hook تم `useThemedStyles` استفاده می‌کند.
- Component Library: هیچ کتابخانه UI شخص ثالث بزرگی دیده نمی‌شود؛ کامپوننت‌ها در `apps/web/src/components` و `apps/mobile/components` به صورت اختصاصی ساخته شده‌اند.
- Animation Library: شواهد از کتابخانه انیمیشن خارجی وجود ندارد؛ انیمیشن‌ها و تعاملات با CSS ساده و React Native استاندارد انجام شده‌اند.
- Theme System: `@castaminofen/ui-tokens` برای وب و موبایل. وب از `data-theme` و `ThemeScript` استفاده می‌کند، موبایل از `SecureStore` و `useColorScheme`.
- Design Tokenها: رنگ‌ها و radius در `packages/ui-tokens/tokens.css` و `packages/ui-tokens/src/tokens.ts` تعریف شده‌اند.
- معماری پوشه‌ها: monorepo با `apps/web`, `apps/mobile`, `packages/shared`, `packages/ui-tokens`, `services/api-gateway`.

# ۴. بررسی سیستم طراحی

- رنگ‌ها: سیستم رنگ مشخص و مستند شده است. پالت اصلی شامل `#776CFE`, `#00EA99`, `#A03CFF`, `#99BE7D` است.
- Typography: فونت فارسی Vazirmatn در وب و موبایل استفاده می‌شود. این مورد همخوانی خوبی با هدف RTL فارسی دارد.
- Spacing: موبایل توکن‌های spacing تعریف شده دارد. در وب فاصله‌ها بیشتر به صورت کلاسیک و ثابت نوشته شده و نشانه‌ای از سیستم spacing یکپارچه در CSS variables دیده نمی‌شود.
- Componentها: کامپوننت‌های پایه مانند `ContentCard`, `EmptyState`, `PlayerBar` و `ThemeProvider` وجود دارند. اما سیستم طراحی کامل برای اجزای مشترک هنوز به صورت واحد ثبت نشده است.
- Layoutها: layout وب با sticky nav و grid کارت‌ها تعریف شده، موبایل با bottom tabs و safe area. این دو ساختار کلی درست هستند.
- Consistency: رنگ و تایپوگرافی نسبتا یکدست است. با این حال، وب و موبایل هنوز به صورت کامل همگرا نیستند و اشکال کمی در یکپارچگی naming و tokenها وجود دارد.

مشکل‌های مشخص:
- عدم وجود spacing tokens یکسان در وب.
- تعریف توکن‌ها در دو فرمت مختلف CSS و JS که همگام‌سازی را سخت می‌کند.
- برخی کلاس‌ها و نام‌ها (مثل `btn-primary`, `nav-link`, `content-card`) از سیستم یکپارچه کلاس‌بندی بهره نمی‌برند.

# ۷. بخش‌های نیمه‌کاره

- `apps/mobile/app/(tabs)/downloads.tsx`: صفحه دانلودها به صورت placeholder است و قابلیت آفلاین/دانلود هنوز فعال نشده.
- `apps/mobile/app/(tabs)/tools.tsx`: کارت‌های coming soon و برخی بخش‌ها فعال نشده‌اند.
- `apps/mobile/app/video/[id].tsx`: بخش «اپیزودهای بیشتر به‌زودی» placeholder دارد.
- `apps/web/src/components/PaywallModal.tsx`: جریان پرداخت وجود دارد اما به نظر می‌رسد برای MVP به صورت stub/شبیه‌سازی شده است.
- `apps/web/src/components/EpisodeUploadForm.tsx`: فرم آپلود آماده است اما پیوند کامل با backend مشخص نیست.
- `apps/mobile/app/content/[id].tsx` و `apps/web/src/app/(main)/content/[id]/page.tsx`: تجربه محتوا پایه دارد ولی برخی حالات خطا و بارگذاری ساده پیاده شده‌اند.
- `packages/ui-tokens`: توکن‌های رنگ و radius هستند اما بین web و mobile کاملاً یکسان نشده‌اند.

# ۸. ابهام‌هایی که فقط صاحب پروژه می‌تواند پاسخ دهد

- آیا `Downloads` و آفلاین در MVP هدف اصلی هستند یا صرفاً فاز ۲؟
- آیا پرداخت پریمیوم موجود در `PaywallModal` قرار است واقعی باشد یا برای نمایش اولیه/شبیه‌سازی است؟
- آیا workflow سازنده باید شامل upload اپیزود، edit chapters و مدیریت کامل محتوا باشد یا فقط content creation و publish؟
- آیا هدف نهایی استفاده از یک design token package کاملاً مشترک برای web و mobile است؟
- معیار کامل‌بودن UI در این پروژه به‌طور دقیق چیست؟ (MVP مقابل Growth)
- آیا نسخه وب قرار است در آینده desktop app یا PWA مستقل شود؟
- آیا APIهای پرداخت و دانلود اکنون در backend پیاده‌سازی شده‌اند یا فقط UI برای آینده آماده شده است؟

# ۹. پیشنهاد قدم بعدی

1. تکمیل و حذف placeholderهای `Downloads` و `Tools` در موبایل. این بخش‌ها شفاف‌ترین نقاط ناکامل UX هستند و تکمیل آنها تجربه کاربری را به صورت محسوس بهبود می‌دهد.
2. یکپارچه‌سازی و مستندسازی دقیق `Design Tokens` بین وب و موبایل. هماهنگ کردن CSS variables و JS tokenها، توسعه آتی UI و طراحی را سریع‌تر و پایدارتر می‌کند.
