# ماموریت: بازیابی وضعیت پروژه — تمرکز روی زیرساخت، ابزار و پیکربندی (نه UI، نه منطق Backend)

تو الان به این پروژه دسترسی داری اما هیچ حافظه‌ای از مکالمات قبلی من با تو نداری.
فرض کن هیچ چیزی درباره این پروژه نمی‌دونی و باید همه چیز رو از خود کد و پیکربندی‌ها کشف کنی.

⚠️ محدوده کار: هر چیزی که نه UI/UX است و نه منطق تجاری بک‌اند (routes, controllers, business logic).
یعنی دقیقاً این‌ها:
- زیرساخت و DevOps (Docker, docker-compose, Kubernetes, فایل‌های deploy)
- CI/CD (GitHub Actions, GitLab CI, پایپ‌لاین‌های build/test/deploy)
- تنظیمات build و bundler (webpack, vite, tsconfig, babel, و غیره)
- مدیریت وابستگی‌ها (package.json/lockfile، نسخه‌های خطرناک یا منسوخ)
- تست‌ها به‌عنوان زیرساخت (نه محتوای تست UI یا بک‌اند، بلکه اینکه اصلاً pipeline تست درست کار می‌کنه یا نه)
- متغیرهای محیطی و مدیریت secrets (.env, .env.example, secret managers)
- امنیت در سطح پروژه (dependency vulnerabilities، تنظیمات CORS، rate limiting در سطح زیرساخت نه منطق)
- مستندسازی پروژه (README، CONTRIBUTING، ADRها، دیاگرام معماری)
- مانیتورینگ، لاگینگ، و ابزارهای observability (اگر تنظیم شده باشه)

به کامپوننت‌های UI و منطق تجاری بک‌اند (مثل قوانین کسب‌وکار داخل controllers) کاری نداشته باش،
مگر برای فهمیدن اینکه یک سرویس یا env variable برای چی استفاده می‌شه.

قبل از پیشنهاد دادن یا نوشتن حتی یک خط کد جدید، مراحل زیر رو دقیقاً انجام بده:

## مرحله ۱ — شناخت زیرساخت پروژه
- فایل‌های Dockerfile، docker-compose.yml، و هر فایل deploy (Vercel/Netlify/Railway/K8s manifests) رو بخون.
- فولدر .github/workflows یا معادل CI دیگه رو بررسی کن و بفهم چه مراحلی (build/test/lint/deploy) اتوماتیک اجرا می‌شن.
- package.json (یا معادلش) رو از نظر اسکریپت‌ها (dev/build/test/lint) و نسخه دیپندنسی‌ها بررسی کن.
- فایل‌های تنظیمات (tsconfig.json، .eslintrc، vite.config، webpack.config) رو مرور کن.
- README و هر مستند معماری (docs/architecture.md یا مشابه) رو بخون.

## مرحله ۲ — تاریخچه کار روی زیرساخت
- `git log --oneline -30` رو بررسی کن و commit های مربوط به فایل‌های config، CI، Docker، dependency را جدا کن.
- `git status` و `git diff` رو چک کن، مخصوصاً روی lockfile ها، فایل‌های CI و env.
- ببین آخرین اجرای CI (اگه قابل مشاهده‌ست) موفق بوده یا fail شده.
- دنبال کامنت‌های `// TODO`, `// FIXME` توی فایل‌های کانفیگ و اسکریپت‌ها بگرد.

## مرحله ۳ — سلامت فعلی زیرساخت
- بررسی کن که `npm install` (یا معادل آن) بدون خطا انجام می‌شه.
- ببین آیا دیپندنسی منسوخ‌شده، ناسازگار، یا با آسیب‌پذیری امنیتی شناخته‌شده وجود داره (`npm audit` یا مشابه).
- بررسی کن Docker build با موفقیت انجام می‌شه (اگه پروژه Dockerized است).
- pipeline تست رو اجرا کن (نه محتوای تک‌تک تست‌ها، بلکه اینکه خود ابزار تست درست ران می‌شه).
- فایل .env.example رو با متغیرهایی که واقعاً در کد استفاده می‌شن مقایسه کن؛ ببین چیزی جا افتاده یا اضافی هست.

## مرحله ۴ — گزارش قبل از هر اقدام
این گزارش رو دقیقاً با همین ساختار بده:

**۱. خلاصه زیرساخت پروژه:** استک ابزار (CI، Docker، bundler، و غیره) در ۲-۳ جمله.
**۲. وضعیت فعلی build/CI:** آیا build/install/CI سالم اجرا می‌شه یا جایی می‌شکنه.
**۳. آخرین کار انجام‌شده روی زیرساخت:** بر اساس commit ها و diff.
**۴. مسائل نیمه‌کاره یا خطرناک:** دیپندنسی منسوخ، env ناقص، پایپ‌لاین شکسته، مستندات قدیمی.
**۵. فرضیه من درباره نقطه توقف:** با ذکر شواهد مشخص.
**۶. سوالات من از تو:** هرچیزی که فقط با تصمیمات قبلی من قابل تأیید هست (مثلاً چرا یک سرویس خاص انتخاب شده).
**۷. پیشنهاد قدم بعدی:** ۱-۲ گزینه مشخص، فقط در محدوده زیرساخت/ابزار/پیکربندی.

⚠️ تا وقتی این گزارش رو ننوشتی و من تأییدش نکردم، هیچ تغییری روی فایل‌های کانفیگ، CI، Docker یا dependency نده.
به کد UI یا منطق تجاری بک‌اند دست نزن. اگه چیزی درباره تصمیمات قبلی مشخص نبود، حدس نزن — بپرس.
