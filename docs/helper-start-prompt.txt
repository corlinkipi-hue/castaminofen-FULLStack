# ماموریت: بازیابی وضعیت پروژه — تمرکز روی UI/UX

تو الان به این پروژه دسترسی داری اما هیچ حافظه‌ای از مکالمات قبلی من با تو نداری.
فرض کن هیچ چیزی درباره این پروژه نمی‌دونی و باید همه چیز رو از خود کد کشف کنی.

⚠️ محدوده کار: فقط لایه UI/UX (کامپوننت‌ها، صفحات، استایل‌ها، تجربه کاربری، دسترسی‌پذیری).
به منطق بک‌اند، API، دیتابیس یا زیرساخت سرور دست نزن و درباره‌شون هم نظر نده مگر اینکه مستقیماً روی UI اثر بذاره
(مثلاً یک state که باعث باگ بصری می‌شه).

قبل از پیشنهاد دادن یا نوشتن حتی یک خط کد جدید، مراحل زیر رو دقیقاً انجام بده:

## مرحله ۱ — شناخت لایه فرانت‌اند
- ساختار پوشه فرانت‌اند رو مرور کن (components/, pages/, styles/, hooks/, و غیره).
- فریمورک UI (React/Vue/...)، کتابخانه استایل (Tailwind/CSS Modules/Styled Components/...) و سیستم کامپوننت (مثل shadcn، MUI) رو شناسایی کن.
- اگه design system یا فایل توکن‌های طراحی (رنگ، فونت، فاصله‌گذاری) وجود داره، پیدا کن و بخون.
- هر فایل Storybook، Figma link، یا مستندات UI رو بررسی کن.

## مرحله ۲ — تاریخچه کار روی UI
- `git log --oneline -30` رو بررسی کن و فقط commit هایی که به فایل‌های UI/استایل مربوطن رو جدا کن.
- `git status` و `git diff` رو چک کن، مخصوصاً روی فایل‌های .css/.tsx/.jsx/.vue.
- دنبال کامنت‌های `// TODO`, `// FIXME` توی کامپوننت‌ها بگرد.
- ببین آیا کامپوننتی نصفه‌کاره مونده (مثلاً prop تعریف‌شده ولی استفاده‌نشده، state بلااستفاده، JSX ناقص).

## مرحله ۳ — سلامت فعلی UI
- پروژه رو (اگه امکانش هست) اجرا کن و صفحات اصلی رو چک کن که رندر می‌شن یا خطا می‌دن.
- به دنبال اینکونسیستنسی بصری بگرد: کامپوننت‌های تکراری با استایل متفاوت، فونت/رنگ‌های خارج از سیستم طراحی.
- وضعیت ریسپانسیو بودن (موبایل/دسکتاپ) و رعایت حداقل‌های دسترسی‌پذیری (کنتراست، فوکوس کیبورد، آلت‌تکست) رو بررسی کن.
- تست‌های مرتبط با UI (اگه هست، مثل Testing Library یا Cypress) رو اجرا کن.

## مرحله ۴ — گزارش قبل از هر اقدام
این گزارش رو دقیقاً با همین ساختار بده:

**۱. خلاصه بخش UI پروژه:** این محصول از نظر واسط کاربری چیه (۲-۳ جمله).
**۲. سیستم طراحی فعلی:** فریمورک، کتابخانه استایل، و اینکه چقدر منسجم رعایت شده.
**۳. آخرین کار UI انجام‌شده:** بر اساس commit ها و diff.
**۴. کارهای نیمه‌کاره UI:** کامپوننت ناقص، استایل ناهماهنگ، صفحات نیمه‌ساز.
**۵. فرضیه من درباره نقطه توقف:** با ذکر شواهد.
**۶. سوالات من از تو:** هرچیزی که نمی‌شه از کد فهمید (مثلاً تصمیم طراحی عمدی بوده یا باگ).
**۷. پیشنهاد قدم بعدی:** ۱-۲ گزینه مشخص، فقط در محدوده UI/UX.

⚠️ تا وقتی این گزارش رو ننوشتی و من تأییدش نکردم، هیچ کد جدیدی ننویس.
به فایل‌های بک‌اند/API دست نزن مگر برای فهمیدن اینکه یک داده از کجا میاد (فقط خواندن، نه ویرایش).

# ماموریت: بازیابی وضعیت پروژه — تمرکز روی Backend

تو الان به این پروژه دسترسی داری اما هیچ حافظه‌ای از مکالمات قبلی من با تو نداری.
فرض کن هیچ چیزی درباره این پروژه نمی‌دونی و باید همه چیز رو از خود کد کشف کنی.

⚠️ محدوده کار: فقط لایه بک‌اند (API، منطق سرور، دیتابیس، احراز هویت، سرویس‌ها، صف‌ها، زیرساخت).
به کامپوننت‌های UI و استایل کاری نداشته باش، مگر برای فهمیدن اینکه فرانت‌اند دقیقاً چه دیتایی از API انتظار داره.

قبل از پیشنهاد دادن یا نوشتن حتی یک خط کد جدید، مراحل زیر رو دقیقاً انجام بده:

## مرحله ۱ — شناخت معماری بک‌اند
- ساختار پوشه بک‌اند رو مرور کن (routes/, controllers/, models/, services/, middlewares/، و غیره).
- فریمورک (Express/NestJS/Django/FastAPI/...) و ORM/دیتابیس مورد استفاده رو شناسایی کن.
- فایل‌های .env.example، config/, و مستندات API (مثل Swagger/OpenAPI) رو بخون.
- schema دیتابیس یا migration ها رو بررسی کن تا مدل داده فعلی رو بفهمی.

## مرحله ۲ — تاریخچه کار روی بک‌اند
- `git log --oneline -30` رو بررسی کن و commit های مربوط به routes/models/services رو جدا کن.
- `git status` و `git diff` رو چک کن، مخصوصاً روی فایل‌های سرور، migration، و config.
- migration های اجراشده و اجرانشده رو مقایسه کن؛ ببین دیتابیس با آخرین schema هماهنگه یا نه.
- دنبال کامنت‌های `// TODO`, `// FIXME`, endpoint هایی که پیاده‌سازی نشدن (مثلاً `res.send("not implemented")`) بگرد.

## مرحله ۳ — سلامت فعلی بک‌اند
- سرور رو (اگه امکانش هست) اجرا کن و ببین بالا میاد یا خطا می‌ده (مخصوصاً اتصال دیتابیس).
- تست‌های backend (unit/integration) رو اجرا کن و نتیجه رو گزارش کن.
- endpoint های موجود رو لیست کن و مشخص کن کدوم واقعاً کار می‌کنن، کدوم نیمه‌کاره‌ن، کدوم فقط تعریف شدن ولی منطق ندارن.
- بررسی کن آیا مسائل امنیتی واضح (مثل کلید هاردکد شده، عدم اعتبارسنجی ورودی) وجود داره.

## مرحله ۴ — گزارش قبل از هر اقدام
این گزارش رو دقیقاً با همین ساختار بده:

**۱. خلاصه معماری بک‌اند:** استک فنی و ساختار کلی (۲-۳ جمله).
**۲. مدل داده فعلی:** خلاصه‌ای از schema/جدول‌ها و روابط اصلی.
**۳. آخرین کار انجام‌شده:** بر اساس commit ها و diff.
**۴. کارهای نیمه‌کاره:** endpoint های ناقص، migration های اجرانشده، تست‌های شکسته.
**۵. فرضیه من درباره نقطه توقف:** با ذکر شواهد مشخص.
**۶. سوالات من از تو:** هرچیزی که فقط با دسترسی به تصمیمات قبلی من قابل تأیید هست.
**۷. پیشنهاد قدم بعدی:** ۱-۲ گزینه مشخص، فقط در محدوده بک‌اند.

⚠️ تا وقتی این گزارش رو ننوشتی و من تأییدش نکردم، هیچ کد جدیدی ننویس و migration جدیدی اجرا نکن.
اگه چیزی درباره تصمیمات معماری قبلی مشخص نبود، حدس نزن — بپرس.
نکته: توی هر دو نسخه، محدودسازی صریح («فقط این لایه، به اون یکی دست نزن») از نفوذ Cursor به حوزه دیگه جلوگیری می‌کنه — این خیلی مهمه چون معمولاً AI ها وقتی یه باگ می‌بینن، وسوسه می‌شن کل استک رو دستکاری کنن حتی وقتی ازشون فقط یه لایه خواسته شده.

# ماموریت: بازیابی وضعیت پروژه — تمرکز روی زیرساخت، ابزار و پیکربندی (نه UI، نه منطق Backend)

تو الان به این پروژه دسترسی داری اما هیچ حافظه‌ای از مکالمات قبلی من با تو نداری.
فرض کن هیچ چیزی درباره این پروژه نمی‌دونی و باید همه چیز رو از خود کد و پیکربندی‌ها کشف کنی.

⚠️ محدوده کار: هر چیزی که نه UI/UX است و نه منطق تجاری بک‌اند (routes, controllers, business logic).
یعنی دقیقاً این‌ها:
- زیرساخت و DevOps (Docker, docker-compose, Kubernetes, فایل‌های deploy)
- CI/CD (GitHub Actions, GitLab CI, پایپ‌لاین‌های build/test/deploy)
- تنظیمات build و bundler (webpack, vite, tsconfig, babel, و غیره)
- مدیریت وابستگی‌ها (package.json/lockfile، نسخه‌های خطرناک یا منسوخ)
- تست‌ها به‌عنوان زیرساخت (نه محتوای تست UI یا بک‌اند، بلکه اینکه اصلاً pipeline تست درست کار می‌کنه یا نه)
- متغیرهای محیطی و مدیریت secrets (.env, .env.example, secret managers)
- امنیت در سطح پروژه (dependency vulnerabilities، تنظیمات CORS، rate limiting در سطح زیرساخت نه منطق)
- مستندسازی پروژه (README، CONTRIBUTING، ADRها، دیاگرام معماری)
- مانیتورینگ، لاگینگ، و ابزارهای observability (اگر تنظیم شده باشه)

به کامپوننت‌های UI و منطق تجاری بک‌اند (مثل قوانین کسب‌وکار داخل controllers) کاری نداشته باش،
مگر برای فهمیدن اینکه یک سرویس یا env variable برای چی استفاده می‌شه.

قبل از پیشنهاد دادن یا نوشتن حتی یک خط کد جدید، مراحل زیر رو دقیقاً انجام بده:

## مرحله ۱ — شناخت زیرساخت پروژه
- فایل‌های Dockerfile، docker-compose.yml، و هر فایل deploy (Vercel/Netlify/Railway/K8s manifests) رو بخون.
- فولدر .github/workflows یا معادل CI دیگه رو بررسی کن و بفهم چه مراحلی (build/test/lint/deploy) اتوماتیک اجرا می‌شن.
- package.json (یا معادلش) رو از نظر اسکریپت‌ها (dev/build/test/lint) و نسخه دیپندنسی‌ها بررسی کن.
- فایل‌های تنظیمات (tsconfig.json، .eslintrc، vite.config، webpack.config) رو مرور کن.
- README و هر مستند معماری (docs/architecture.md یا مشابه) رو بخون.

## مرحله ۲ — تاریخچه کار روی زیرساخت
- `git log --oneline -30` رو بررسی کن و commit های مربوط به فایل‌های config، CI، Docker، dependency را جدا کن.
- `git status` و `git diff` رو چک کن، مخصوصاً روی lockfile ها، فایل‌های CI و env.
- ببین آخرین اجرای CI (اگه قابل مشاهده‌ست) موفق بوده یا fail شده.
- دنبال کامنت‌های `// TODO`, `// FIXME` توی فایل‌های کانفیگ و اسکریپت‌ها بگرد.

## مرحله ۳ — سلامت فعلی زیرساخت
- بررسی کن که `npm install` (یا معادل آن) بدون خطا انجام می‌شه.
- ببین آیا دیپندنسی منسوخ‌شده، ناسازگار، یا با آسیب‌پذیری امنیتی شناخته‌شده وجود داره (`npm audit` یا مشابه).
- بررسی کن Docker build با موفقیت انجام می‌شه (اگه پروژه Dockerized است).
- pipeline تست رو اجرا کن (نه محتوای تک‌تک تست‌ها، بلکه اینکه خود ابزار تست درست ران می‌شه).
- فایل .env.example رو با متغیرهایی که واقعاً در کد استفاده می‌شن مقایسه کن؛ ببین چیزی جا افتاده یا اضافی هست.

## مرحله ۴ — گزارش قبل از هر اقدام
این گزارش رو دقیقاً با همین ساختار بده:

**۱. خلاصه زیرساخت پروژه:** استک ابزار (CI، Docker، bundler، و غیره) در ۲-۳ جمله.
**۲. وضعیت فعلی build/CI:** آیا build/install/CI سالم اجرا می‌شه یا جایی می‌شکنه.
**۳. آخرین کار انجام‌شده روی زیرساخت:** بر اساس commit ها و diff.
**۴. مسائل نیمه‌کاره یا خطرناک:** دیپندنسی منسوخ، env ناقص، پایپ‌لاین شکسته، مستندات قدیمی.
**۵. فرضیه من درباره نقطه توقف:** با ذکر شواهد مشخص.
**۶. سوالات من از تو:** هرچیزی که فقط با تصمیمات قبلی من قابل تأیید هست (مثلاً چرا یک سرویس خاص انتخاب شده).
**۷. پیشنهاد قدم بعدی:** ۱-۲ گزینه مشخص، فقط در محدوده زیرساخت/ابزار/پیکربندی.

⚠️ تا وقتی این گزارش رو ننوشتی و من تأییدش نکردم، هیچ تغییری روی فایل‌های کانفیگ، CI، Docker یا dependency نده.
به کد UI یا منطق تجاری بک‌اند دست نزن. اگه چیزی درباره تصمیمات قبلی مشخص نبود، حدس نزن — بپرس.

# ماموریت: بازیابی وضعیت پروژه — تمرکز روی Testing/QA

تو الان به این پروژه دسترسی داری اما هیچ حافظه‌ای از مکالمات قبلی من با تو نداری.
فرض کن هیچ چیزی درباره این پروژه نمی‌دونی و باید همه چیز رو از خود کد و تست‌ها کشف کنی.

⚠️ محدوده کار: فقط لایه تست و کیفیت (QA). یعنی دقیقاً این‌ها:
- تست‌های واحد (Unit Tests)
- تست‌های یکپارچگی (Integration Tests)
- تست‌های end-to-end (Cypress, Playwright, Selenium و مشابه)
- پوشش تست (Test Coverage) و گزارش‌های مرتبط
- Mock ها، Fixture ها، و داده‌های تست
- ابزارهای لینت/فرمت که به کیفیت کد مربوطن (ESLint, Prettier, type-checking)
- استراتژی تست (آیا TDD رعایت شده، چه لایه‌هایی تست دارن و چه لایه‌هایی ندارن)

به این‌ها کاری نداشته باش (فقط بخون، تغییر نده):
- منطق تجاری واقعی در UI یا Backend (مگر برای فهمیدن اینکه چرا یک تست می‌شکنه)
- تنظیمات CI/CD خود پایپ‌لاین (فقط بخشی که مربوط به اجرای تست‌هاست را بررسی کن، نه کل workflow)
- Docker/deploy/infra

قبل از پیشنهاد دادن یا نوشتن حتی یک خط کد جدید، مراحل زیر رو دقیقاً انجام بده:

## مرحله ۱ — شناخت زیرساخت تست
- فولدرهای تست (tests/, __tests__/, spec/, cypress/, e2e/) رو پیدا کن و ساختارشون رو مرور کن.
- فریمورک‌های تست استفاده‌شده رو شناسایی کن (Jest, Vitest, Mocha, Pytest, Cypress, Playwright و غیره).
- تنظیمات مربوطه (jest.config, vitest.config, playwright.config) رو بخون.
- ببین آیا ابزار اندازه‌گیری پوشش تست (coverage) فعال هست و آخرین گزارش پوشش (اگه موجوده) چی می‌گه.

## مرحله ۲ — تاریخچه کار روی تست‌ها
- `git log --oneline -30` رو بررسی کن و commit های مربوط به فایل‌های تست را جدا کن.
- `git status` و `git diff` رو چک کن، مخصوصاً روی فایل‌های *.test.*, *.spec.*
- ببین آیا تستی وجود داره که نوشته شده ولی skip شده (`it.skip`, `describe.skip`, `@pytest.mark.skip`) یا commented-out هست.
- دنبال کامنت‌های `// TODO: add test`, `// FIXME` مرتبط با تست بگرد.

## مرحله ۳ — سلامت فعلی تست‌ها
- کل مجموعه تست رو اجرا کن و نتیجه رو دقیق ثبت کن: چند تست pass، چند fail، چند skip.
- برای هر تست fail شده، دلیل رو بررسی کن: آیا خود تست خرابه، یا کد اصلی تغییر کرده و تست به‌روز نشده؟
- بررسی کن آیا تست‌های flaky (گاهی pass گاهی fail بدون تغییر کد) وجود داره.
- نسبت پوشش تست رو در بخش‌های مختلف پروژه مقایسه کن (کجا تست زیاد داره، کجا اصلاً تست نداره).
- بررسی کن ابزارهای لینت/type-check بدون خطای جدید اجرا می‌شن.

## مرحله ۴ — گزارش قبل از هر اقدام
این گزارش رو دقیقاً با همین ساختار بده:

**۱. خلاصه استراتژی تست پروژه:** چه فریمورک‌هایی و چه سطحی از تست (unit/integration/e2e) وجود داره (۲-۳ جمله).
**۲. وضعیت فعلی اجرای تست‌ها:** تعداد pass/fail/skip و اینکه کدوم بخش‌ها شکسته‌ن.
**۳. آخرین کار انجام‌شده روی تست‌ها:** بر اساس commit ها و diff.
**۴. شکاف‌های پوشش تست:** کدوم بخش‌های کد اصلاً تست ندارن یا تست‌هاشون ناقصه.
**۵. فرضیه من درباره نقطه توقف:** با ذکر شواهد مشخص (مثلاً «آخرین commit یه تست جدید اضافه کرده ولی fail می‌شه، احتمالاً نصفه‌کاره مونده»).
**۶. سوالات من از تو:** هرچیزی که فقط با دسترسی به تصمیمات قبلی من قابل تأیید هست (مثلاً چرا یک بخش عمداً تست نداره).
**۷. پیشنهاد قدم بعدی:** ۱-۲ گزینه مشخص، فقط در محدوده تست/QA.

⚠️ تا وقتی این گزارش رو ننوشتی و من تأییدش نکردم، هیچ تست جدیدی ننویس و تست موجود رو تغییر نده.
اگه یک تست شکسته به نظر می‌رسه ولی مطمئن نیستی که باگ کده یا تست منسوخه، حدس نزن — توی بخش سوالات بپرس.
نکته: حالا چهار لایه کامل شد: UI/UX، Backend، Infra/DevOps/Tooling، و Testing/QA. توی نسخه Infra قبلی گفته بودم «محتوای تک‌تک تست‌ها» رو بررسی نکنه فقط سلامت پایپ‌لاین رو چک کنه — این نسخه دقیقاً همون شکاف رو پر می‌کنه. اگه می‌خوای، می‌تونم یه پرامپت «راهنمای انتخاب» هم بسازم که بهت بگه بسته به وضعیت پروژه کدوم یک (یا کدوم ترکیب) از این ۴ نسخه رو باید بفرستی.

تدوین راهنمای تصمیم‌گیری برای انتخاب نسخه مناسب
این‌هم راهنمای انتخاب — یه چک‌لیست کوتاه که بر اساس علامت‌های وضعیت پروژه، بهت می‌گه کدوم نسخه (یا ترکیبی از نسخه‌ها) رو باید بفرستی:

راهنمای انتخاب نسخه مناسب
۱. اگه نمی‌دونی دقیقاً کجای پروژه موندی → نسخه عمومی (نسخه اول)
وقتی از یه پروژه خیلی وقته دور بودی و حتی حوزه کاری آخرین‌بار (UI بوده یا Backend یا چیز دیگه) رو هم یادت نیست. این نسخه یه دید کلی از کل پروژه می‌ده و بهت کمک می‌کنه بفهمی از کجا شروع کنی — حتی می‌تونی از گزارشش استفاده کنی تا بعداً تصمیم بگیری کدوم لایه‌محور رو بفرستی.

۲. اگه یادته آخرین‌بار داشتی رو ظاهر/تجربه کاربری کار می‌کردی → نسخه UI/UX
نشونه‌هاش: آخرین commit هات اسم فایل‌های .tsx/.css/.vue دارن، یا یادته داشتی یه صفحه یا کامپوننت خاص رو می‌ساختی.

۳. اگه یادته داشتی رو API/منطق سرور/دیتابیس کار می‌کردی → نسخه Backend
نشونه‌هاش: آخرین commit هات اسم فایل‌های routes/controllers/models دارن، یا یادته داشتی یه endpoint یا migration جدید اضافه می‌کردی.

۴. اگه مطمئن نیستی پروژه اصلاً درست run/build می‌شه یا نه → نسخه Infra/DevOps
نشونه‌هاش: آخرین‌بار که باز کردی پروژه، یا CI قرمز بود، یا npm install خطا داد، یا یادت میاد داشتی یه چیزی رو تو Docker/CI درست می‌کردی.

۵. اگه یادته داشتی رو تست‌ها کار می‌کردی یا پوشش تست رو بالا می‌بردی → نسخه Testing/QA
نشونه‌هاش: آخرین commit هات اسم فایل‌های .test./.spec. دارن، یا یادته یه باگ رو داشتی با نوشتن تست رهگیری می‌کردی.

۶. اگه اصلاً هیچی یادت نیست (نه حوزه، نه وضعیت) → همیشه از نسخه عمومی شروع کن
هیچ‌وقت مستقیم سراغ یه نسخه لایه‌محور نرو اگه مطمئن نیستی کدوم لایه. نسخه عمومی گزارشی می‌ده که خودش بهت می‌گه آخرین کار مال کدوم لایه بوده — بعد با اون اطلاعات، نسخه دقیق‌تر رو برای همون لایه بفرست.

۷. اگه پروژه چند لایه هم‌زمان درگیره (مثلاً یه فیچر که هم UI هم Backend داره)
اول نسخه عمومی رو بفرست تا تصویر کلی بگیری، بعد به‌ترتیب اولویت (معمولاً Backend اول چون UI بهش وابسته‌ست) نسخه‌های تخصصی رو جدا جدا بفرست — هیچ‌وقت دو نسخه لایه‌محور رو هم‌زمان نفرست، چون Cursor گزارش‌ها رو قاطی می‌کنه و محدودیت "دست نزدن به لایه دیگه" رو نقض می‌کنه.

قانون کلی: نسخه عمومی = نقطه شروع پیش‌فرض وقتی مطمئن نیستی. نسخه‌های لایه‌محور = فقط وقتی حداقل یه سرنخ (از commit، از حافظه خودت، یا از گزارش نسخه عمومی) داری که دقیقاً کدوم بخش رو باید بررسی کنه.
